Literature survey

Literature review of the existing Adaptive authentication technology.

Risk-based authentication is still in performance level. Adaptive authentication's one of the major core components to layer up is Behavioral biometrics, current researches have been conducted to discover the activity patterns of the user. Based on the activity patterns of users, researchers have identified these as unique signatures. This uniqueness in the user’s behavior or the activity patterns of a user has shown better qualities to qualify as a better candidate in software authentication. These user patterns have been developed to be utilized in an adaptive manner in authentication, whereas these adaptive signatures are fed to the authentication in the process of decision making which uses statistical computational models. Activity patterns of the user include mouse and keystroke dynamics which aids in the context of authentication. User behavior, on the other hand, is sort of a sophisticated intricacy. Context perception should be a fact to attain when understanding the relationship between users, devices, and contexts. Furthermore, mobile devices are trending to be among the factors in authentication whereas it plays a major role by being in modern life as a piece of mandatory equipment. As smartphone devices are capable of rendering complicated resources such as the exact location of the user, biometrics such as fingerprint, facial recognition. Not limited to biometrics and geo-locations of the user, device recognition of a respective user has shown a gradual improvement in research areas where this factor could possibly be included in authentication contexts. Application usage based on the platform of the user can also be used as a matrix when checking the authentication against the device.

Apart from the user behavior and device authentication the other factor that has been considered in adaptive authentication is geolocation. In the current setting of authentication mechanisms, the location of the user has been taken into consideration when attempting to provide access to the system. In such a scenario, the other revealing element is that the system can keep track of the user login attempts and check the geolocation of the current attempt with compared to the previous which will have certain lockout mechanism for the account if any discrepancies were found: past logins records of the users are maintained in the baseline profile of the user. All the above mechanisms used above are used and forwarded to the Risk engine to calculate the risk score of the user related to the login attempt. Whereas the user's behavior, location, and the device used to log into the system and other factors are being considered from this stage of the authentication process. As mentioned above the authentication server will keep records of the users as a user profile and this user profile will be cross-checked against the properties shared by the user at the instance and compare the resultant score from the risk engine and then decides whether the user is given the access or not. Once the user enters the credentials in a correct manner, the risk engine will activate.

Research Methodology

The way AuthDNA Adaptive authentication service built

As declared above the adaptive authentication mechanism comprised of four major components such that the authentication mechanism will be distributed under these components to gather information and authenticate the user in different contexts. Namely,
-Behavior-Based Authentication.
-Device and Browser-Based Authentication.
-Geo-Location And Network-Based Authentication.
-Risk-Based Authentication.
In each of the components, the authentication mechanism will be distributed and will transpire individually. As in each of the components, the user shared properties will be determined. Afterward, by undergoing a series of processes each of the components will predict an accuracy value as a percentage. The resultant accuracy value predicted by each of the authentication components will then send it to the risk-based authentication component for further assessing the authentication request. By assessing the user parameter values sent by each of the other authentication components, risk-based authentication will proceed with deep learning algorithms to deduce the legitimacy of the request. With a thorough examination, risk-based authentication will conclude in granting the permission or proceed with re-authentication by reassuring the permission for access.

Behavior-Based
To create a reliable user profile the AuthDNA adaptive authentication service will use the keystroke patterns (keystroke dynamics). This adaptive authentication service will be available for identity servers. basically, identity servers don't have a front end. The front end is users own. identity servers get the inputs from these user own front ends through rest APIs. in this case, AuthDNA adaptive authentication service needs to gather the keystroke data from user own front end. so, the mechanism of gathering keystroke is portable. there is a JavaScript class developed and need to implement into the user own front end. then need to create an object from the JavaScript class and set the targets of where to gather the keystroke data. mostly it will be username and password input fields of the front end. then they will analysis at the Deep learning model. Deep learning module used is LSTMs which a customized RNN which is a neural network. neural networks are better for the classification of keystroke data. they will increase the accuracy other than machine learning. LSTM model was used with Nadam optimizer with the learning rate of 0.002, Beta one value of 0.9, beta two value of the 0.999 the schedule decay vale of 0.004 and Softmax activation function for increasing the accuracy.

Device and Browser-Based
Under device-based authentication, it introduces,
-A Mobile Application.
-A Browser Plugin.
Mobile application in the authentication component will help in identifying the authentication modalities of the mobile device used by the user while determining a fingerprint for the device. Upon registration, users are advised to download the mobile application and use the registration section to register the mobile device under the profile. To allow the registration to process the user must be authenticated priory.Once the mobile application is installed, and registered under the user, the mobile application will try to collect the details of the device and generates a fingerprint for the device which is then stored in the system for further analysis.
A browser plugin will be created in order to extract information from the user device’s browser that uses to login to the system in order to create a unique identifier for the browser. As mentioned earlier, the traditional method can bypass using several advanced techniques.Upon each authentication request, the authentication for the device will be carried out wholly by the mobile application. Whilst in the absence of the mobile application, the process will be taken over and carried out by the middleware of the system and will forward an accuracy rate as a percentage to the risk-based authentication for further processing.

Geo-Location And Network-Based
Wherever the user wishes to log in to their account, they will first open the application, enter their username and password, and submit this to the local server. The Mail Server then receives this information. The Mail Server will validate these credentials and the user will be requested to scan his thumb if they are right. This next stage is the scan thumb. The local server validates this fingerprint impression regionally and then tracks the location data. This device offers Local Server with the user's space-time data, i.e. latitude & longitude. The Local Server stores all user information such as username, password, fingerprint template, latitude, longitude, and range of that place from its preferred place. The local server will establish a connection between the User and the Mail Server after successful login, and after that, the user would be able to compile emails, send emails and check inbox. All of these user details (such as username, password, e-mail details) are stored in an encrypted format on the server. Using the AES algorithm, the sending and receiving signals are also encrypted. The user coverage area is specified. If a user leaves that area after successful login, they will be prohibited from accessing their account.

Risk-Based
In Risk-Based Authentication, the request propagated from the user will be assessed further with properties shared by the user for the authentication instance, including the context. The architectural diagram below shows where the data will be gathered and the behavior of the authentication procedure.According to the users in the system, Maintain a Risk Profile for the user aligning the risk related to the user by considering facts like privileges, roles and etc. Furthermore, data transferred from other authenticators factors such as Behavior, Geo-Location and Networks, and the Device & Browser-based authentication are fed to the risk engine for further calculations in risk scores.When a user tries to log in to the system, consider the Risk Profile and find out the anomalies and discrepancies of credentials user share at the instance by gathering data from different authentication mechanisms. Afterward, using Machine Learning Algorithms, calculate a risk score and provide MFA as the re-authentication mechanism to challenge the user according to the risk score obtained in the calculation phase. The model used in the calculation of the risk score is the Risk Calculation Model which take in 3 machine learning algorithms such as SVM, One-Class SVM, Naive Bayesian.
